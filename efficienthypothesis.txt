OVERVIEW
--------------------------------------------------------------------------------
Website:      efficienthypothesis.com
Purpose:      Personal productivity platform. Users log in with Google, manage
              tasks (tasks and routines, formerly one-off and recurring) with time tracking, set goals with
              daily data logging + chart visualization, and view weekly/monthly
              calendars of completed work. All user data is per-account.

              Lightweight — NO pdflatex, NO TeX Live, NO Asymptote.
              ZIP-based Lambda (not container image). Deploys in ~10 seconds.

              (may need to be restored from the deploy zip in S3 between
              sessions — see DEPLOYMENT section).

  original problemanalytics.com monolith. Shares the same DynamoDB Users
  table and Google OAuth client ID with problemanalytics.com, but has its
  own Lambda, API Gateway, S3 bucket, and domain.

TIMEZONE HANDLING
--------------------------------------------------------------------------------
Auto-detect: On first authenticated page load, if the user has no timezone
  set in DynamoDB, the browser's Intl.DateTimeFormat().resolvedOptions().timeZone
  is auto-saved silently (no setup screen).

Storage: UTC everywhere in the database with 'Z' suffix. The timezone field on
  the Users table is used for display conversion only.

CRITICAL: ensureUTC(iso) — Frontend helper that appends 'Z' to bare ISO
  strings. Applied to all 5 parsing functions: utcToLocalDate, getLocalHourFrac,
  formatTime, formatDateTime, toLocalDatetimeValue. Without this, JS Date()
  treats bare ISO strings as local time, causing tasks to appear hours off.

  TEST CASE: A task created at 8:30 PM Pacific (03:30 UTC) must render at
  8:30 PM, not 3:30 AM the next day. If times are wrong by the UTC offset,
  ensureUTC is not being applied.

--------------------------------------------------------------------------------
ROUTINE SYSTEM
--------------------------------------------------------------------------------
Two-part architecture: S3 templates + DynamoDB task instances.

TEMPLATES (S3):
  Stored at s3://efficienthypothesis.com/<email>/routines/tasks.json
  Schema:
    {
      "id": "uuid",
      "name": "Wake up",
      "assign_time": "07:00",        // HH:MM — when task becomes visible
      "due_time": "07:30",           // HH:MM — deadline
      "first_day": "2026-05-01",     // YYYY-MM-DD — first instance date
      "pattern": "interval:1",       // see PATTERN FORMAT below
      "max_instances": 62,           // total instances before deactivation
      "instances": 5,                // counter, incremented each materialization
      "folder": "folder-path or null", // project folder (set via drag-and-drop or creation)
      "active": true,                // false when instances >= max_instances
      "created_at": "2026-04-29T...Z"
    }

PATTERN FORMAT:
  Two types, stored as strings:
    "interval:N"    — every N days (N=1 for daily, max 30)
    "set:0,1,3"     — specific weekdays (0=Mon, 1=Tue, ..., 6=Sun)

  Conversions from natural language:
    "daily"                 → interval:1
    "weekdays"              → set:0,1,2,3,4
    "weekends"              → set:5,6
    "every 3 days"          → interval:3
    "every monday"          → set:0
    "every mon wed fri"     → set:0,2,4
    All 7 days in a set     → interval:1 (auto-simplified)
    N > 30                  → clamped to 30

  Display format (patternToDisplay):
    interval:1    → "Daily"
    set:5,6       → "Weekends"
    set:0,1,2,3,4 → "Weekdays"
    set:0,2,4     → "{M, W, F}"
    interval:3    → "Every 3 days"

  Backend matcher (_pattern_matches_date):
    interval: (check_date - first_day).days % N == 0
    set: check_date.weekday() in days  (Python 0=Mon...6=Sun)

  JS matcher (patternMatchesDateJS):
    Same logic but converts JS getDay() (0=Sun...6=Sat) to Python convention.

MATERIALIZATION (POST /api/routines/materialize):
  Runs once on initApp(). For each active template:
  1. Check instances < max_instances
  2. Check today >= first_day
  3. Check today matches pattern via _pattern_matches_date
  4. Check no instance already exists for today (by routine_id + date)
  5. Create DynamoDB task: assign_datetime = today+assign_time (UTC),
     due_datetime = today+due_time (UTC), routine_id = template id
  6. Increment instances counter on S3 template
  7. If instances >= max_instances, set active=false
  Also cleans stale incomplete routine instances from previous days.

ACCESSIBLE DATE RANGE:
  first_day must be: today or later, AND before the forward boundary.
  Forward boundary = last day of (current_month + 2).
    April → June 30, October → December 31, December → Feb 28/29
  max_instances is auto-clamped to the maximum possible within this range.

  TEST CASES:
    4/5  → boundary 6/30
    10/15 → boundary 12/31
    12/31 → boundary 2/28 (or 2/29 leap year)
    12/28 → boundary 2/28 (or 2/29 leap year)

FIRST DAY AUTO-PUSH (set patterns):
  If first_day doesn't match a day in the set, it's silently pushed forward
  to the next matching day.
  TEST CASE: pattern={R,F,S} (Thu,Fri,Sat), first=today (Wednesday 4/29)
    → pushed to Thursday 4/30.
  TEST CASE: pattern=every saturday, first=today (Tuesday 5/5)
    → pushed to Saturday 5/9.

INSTANCES (task instances in DynamoDB):
  Created by materialization. Regular tasks with routine_id set.
  Stale instances (incomplete from previous days) are deleted on each
  materialization. Completed instances stay forever.

--------------------------------------------------------------------------------
TASKS (formerly "One-off Tasks")
--------------------------------------------------------------------------------
Same DynamoDB Tasks table, no routine_id. Schema:
  task_id, user, name, path ("/"), assign_datetime (UTC), due_datetime (UTC),
  end_datetime (UTC or null), time_log [{start, end}], due_status, created_at

Time tracking: start/pause creates time_log entries. Starting auto-pauses any
  other active task. Completing closes open time_log. Weekly calendar visualizes
  these sessions as colored cards.

Subtask nesting: path-based hierarchy. Root = "/". Subtask of "Parent" =
  "/Parent/". Drag-and-drop on frontend. Backend just stores/retrieves path.

DRAFTS:
  Created server-side on modal open (Gmail-style). Stored in the dedicated
  DynamoDB Drafts table (partition key: draft_id). Auto-save on 2s debounce.
  All routine fields (assign_time, first_day, pattern, max_instances) persist
  across sessions. Empty drafts (no name/input) are deleted on
  dismiss. Saving a routine draft: POSTs new S3 template + deletes draft.
  Saving a task draft: POSTs new task to Tasks table + deletes draft.

  TEST CASE: Open routine modal, fill assign_time + pattern + first_day,
    dismiss, reopen from Drafts section → all fields should be restored.

ACTIONS & SCHEDULES
--------------------------------------------------------------------------------
Two new item types complementing Tasks and Routines:

TYPE SYSTEM:
  Base type + repeat:
    Task      + repeat → Routine    (what you NEED to do)
    Action    + repeat → Schedule   (what you DID / ARE DOING)

  Actions track time blocks — they only have start_datetime and end_datetime.
  No assign/due, no time_log, no due_status, no subtask nesting.
  Actions appear ONLY on the weekly calendar, never on tasks page or monthly.

ACTIONS (DynamoDB "Actions" table):
  Partition key: action_id (String) — UUID
  Fields:
    action_id, user (email), name, start_datetime (UTC ISO with Z),
    end_datetime (UTC ISO with Z), schedule_id (UUID or null),
    folder (path or null), is_planned (Boolean), created_at (UTC ISO with Z)

  is_planned = true: dashed-outline card on weekly (from schedule materialization)
  is_planned = false: solid card on weekly (manually created or manifested)

SCHEDULES (S3 templates):
  Stored at s3://efficienthypothesis.com/<email>/schedules/actions.json
  Schema mirrors routines:
    {
      "id": "uuid",
      "name": "Go to school",
      "start_time": "08:00",       // HH:MM — when action starts
      "end_time": "15:00",         // HH:MM — when action ends
      "first_day": "2026-05-06",
      "pattern": "set:0,1,2,3,4",  // same format as routines
      "instances": 0,
      "max_instances": 85,          // OR end_date (mode-based, same as routines)
      "folder": null,
      "active": true,
      "created_at": "2026-05-06T...Z"
    }
  Overnight actions: if end_time <= start_time, end is next day.

MATERIALIZATION (POST /api/schedules/materialize):
  Runs on initApp() alongside routine materialization.
  Creates planned actions (is_planned=true) for today from active templates.
  Also cleans expired planned actions: is_planned=true AND
    end_datetime + 2 days < now → deleted.

MANIFESTATION (POST /api/actions/<id>/manifest):
  Converts a planned action to a real action (is_planned → false).
  RESTRICTION: Can only manifest starting 5 minutes before start_datetime.
  Before that window, returns 400 error.

WEEKLY CALENDAR RENDERING:
  Planned actions: dashed border, transparent background, folder color for
    border/text. Clickable → calls manifestAction(id).
  Manifested/manual actions: solid background (same as task session cards).
  Both positioned by start/end fracs using same HOUR_PX math as task sessions.

CONTEXT MENU:
  "QuickAdd" renamed to "Task" (icon: task_alt).
  New "Action" entry (icon: schedule) opens the Action card.
  Action card has same +/1/2 mode selector for creating Schedules.

API ENDPOINTS:
  GET/POST     /api/actions
  PUT/DELETE   /api/actions/<id>
  POST         /api/actions/<id>/manifest
  GET/POST     /api/schedules
  PUT/DELETE   /api/schedules/<id>
  POST         /api/schedules/materialize

JS STATE:

SMART INPUT PARSING
--------------------------------------------------------------------------------
The text modal in the task creation form parses comma-separated segments:

  For BOTH tasks and routines:
    'name with apostrophe's',  → name (closing quote + comma required)
    Enter key on empty input   → triggers save

  For TASKS (non-routine):
    assign tomorrow,           → assign_datetime (default time 00:00)
    assign thursday,           → this Thursday (or today if Thursday)
    assign next friday,        → next week's Friday
    assign next next monday,   → Monday two weeks out
    assign tomorrow 5 pm,      → assign_datetime with specific time
    assign 16:00,              → today at 4 PM (military time)
    due sunday,                → due_datetime (default time 23:59)
    due friday 10:00 am,       → due_datetime with specific time

  For ROUTINES:
    assign 7:00 am,            → assign_time (12hr format)
    assign 16:00,              → assign_time (24hr/military)
    due 10:00 pm,              → due_time
    due 19:00,                 → due_time (military)
    first today,               → first_day (auto-pushed for set patterns)
    first tomorrow,            → first_day
    first 5/5,                 → May 5 of current year
    first 5/5/2026,            → explicit date
    first thursday,            → next Thursday
    first next monday,         → next week's Monday
    daily,                     → pattern interval:1
    weekdays,                  → pattern set:0,1,2,3,4
    weekends,                  → pattern set:5,6
    every mon wed fri,         → pattern set:0,2,4
    every 3 days,              → pattern interval:3
    every 31 days,             → clamped to interval:30
    5 times,                   → max_instances (only if pattern is set)
    n times,                   → auto-compute maximum instances
    1 times,                   → clamped to 2 (minimum)
    10000 times,               → clamped to max possible

  Day name modifiers: {this, next, next next, null (same as this)}
    "assign tuesday" on Wednesday = next Tuesday (6 days, not past)
    "assign wednesday" on Wednesday = today

  TEST CASES:
    'shave my cat's hair',     → name: "shave my cat's hair" (apostrophe OK)
    first today, (with pattern {S}) → pushed to Saturday (not today)
    daily, first tomorrow, n times, → instances = ~62 (days to boundary)
    every 5 days, 10000 times, → clamped to max within boundary

--------------------------------------------------------------------------------
SPA ARCHITECTURE
--------------------------------------------------------------------------------
All authenticated routes serve app.html. Tab navigation via JavaScript.

Initialization: initApp() → initTimezone() → materialize + fetchAllData()
  → navigateTo(initialPage). Error handling: catch blocks still navigate
  even if data loading fails.

Navigation: navigateTo() clears nowLineInterval, blurs active element,
  toggles flush CSS for tasks/weekly/monthly, renders content, pushes URL,
  updates sidebar active state, renders from cache.

Caching: fetchAllData() loads tasks/drafts/routines/goals/calendar in
  parallel. Tab switches render from cache (zero API calls). Mutations call
  refreshData() which re-fetches and re-renders.

Arrow keys: Left/Right = prev/next week (weekly) or month (monthly).
  Up/Down = scroll weekly view. Only when not in input/textarea/select.
  Sidebar links blur on navigation so arrow keys work immediately.

COMMON GOTCHAS / LESSONS LEARNED
--------------------------------------------------------------------------------
1. UTC TIMESTAMPS — All times UTC with 'Z' suffix in DB. Frontend uses
   ensureUTC() to append 'Z' to bare ISO strings. Backend uses
   datetime.utcnow().isoformat() + 'Z'. Without 'Z', JS Date() treats
   strings as local time, causing hours-off display bugs.

2. CONST/VAR CONFLICTS — In JS event listener functions, using const at
   the top and var later in the same function causes SyntaxError (var hoists
   to function scope, conflicting with const). Use var consistently inside
   event listeners, or use unique variable names. This bug has crashed the
   entire JS file multiple times — always check paren/brace balance.

3. FLUSH LAYOUT — Tasks, Weekly, and Monthly pages use .page-flush class.
   app.html inline <style> uses !important to override base.css/navbar2.css
   padding. #app-content needs height:100% for flush pages. If adding a
   new flush page, add to the condition in navigateTo().

4. WEEKLY FIRST LOAD — renderWeekView() must call initWeekStart() at the
   top. Without it, weekCalStart is null → new Date(null) = Jan 1 1970 →
   all days marked inaccessible → grey grid with no headers.

5. PATTERN FORMAT — Always "interval:N" or "set:X,Y,Z". No legacy string
   formats. _normalize_pattern was removed. Python weekday: 0=Mon...6=Sun.
   JS getDay: 0=Sun...6=Sat. Always convert between the two.

6. DRAFTS TABLE — Drafts live in a separate DynamoDB "Drafts" table
   (partition key: draft_id). The Tasks table has no draft records.
   Frontend uses /api/drafts endpoints for all draft CRUD operations.

7. DRAFT vs TEMPLATE — When resuming a routine draft and saving, the code
   must POST (create new S3 template), not PUT (which tries to update a
   non-existent template). Check: isEditingTemplate = editId && !draftId.

8. PIP DEPENDENCIES — Python packages are provided by the shared
   "python-web-deps" Lambda Layer. Do NOT pip install into the source
   directory. If you need to add/update packages, edit requirements.txt
   and run deploy_layer.sh. The old approach of bundling packages in
   the ZIP caused it to balloon to ~22 MB and was fragile (macOS
   binaries cause "invalid ELF header" on Lambda).

9. SED DANGER — Never use sed to remove function calls with parentheses.
   Greedy regex like [^)]*[)] matches through nested parens and destroys
   surrounding code. Always use targeted Edit operations instead.

10. SIDEBAR FOCUS — sidebar links have outline:none to prevent blue focus
    ring. navigateTo() calls document.activeElement.blur() so arrow keys
    work immediately after clicking a tab.

11. NOW LINE — Uses recursive setTimeout (not setInterval) synced to minute
    boundaries: delay = (60 - currentSeconds) * 1000. Self-corrects drift.
    Must clearTimeout on tab navigation and re-render on day change.

12. INSTANCES CLAMPING — Instances input auto-clamps on keystroke via
    clampInstances(). Max is set by autoComputeMaxIfReady() which runs on
    every pattern/first_day change. Input disabled until pattern is set.

RESTRICTION SYSTEM (PLANNED)
--------------------------------------------------------------------------------
Some hardcoded limits should only apply to manual user entry, not to
programmatic requests from chatbots (in-website chatbot or external MCP
chatbot like GPT/Claude). The purpose is to force users not to overplan
and focus on the present when manually entering data, while allowing
chatbots full flexibility to manage data on the user's behalf.

IMPLEMENTATION:
  Programmatic requests include a flag in the request body:
    { "_programmatic": true, ...other fields... }
  Manual entry (QuickAdd, smart modal, NoteAdd) never includes this flag.
  Both the in-website chatbot and external MCP server set it.

  Backend helper:
    def _is_manual_entry(data):
        return not data.get("_programmatic", False)

  Validation functions accept a strict parameter:
    err = _validate_date_range(dt_str, strict=_is_manual_entry(data))

  Security note: A savvy user could add _programmatic via dev tools to
  bypass their own restrictions. This is acceptable — the restrictions
  are for the user's own productivity benefit, not security enforcement.

RESTRICTIONS — MANUAL-ONLY (skip for chatbots):
  Routine interval range (1-30 days)   app.html:113, productivity.js
  Forward boundary (+2 months)         app.py:928-936

RESTRICTIONS — APPLY TO ALL (structural/system limits):

NOTE DATE INPUT (PLANNED REDESIGN):
  Current: MM/DD/YYYY segmented inputs with maxlength on each segment.
  Planned: Smart auto-tabbing based on value, not character count.
    - Month: if user types 3-9, auto-tab immediately (no two-digit month
      starts with 3+). If user types 1, wait for second digit (10, 11, 12).
      If user types 0, wait for second digit (01-09).
    - Day: if user types 4-9, auto-tab immediately (no two-digit day
      starts with 4+). If user types 1-3, wait for second digit.

      digits entered, input is complete.
    - Backend validates the resulting date is real (e.g., rejects Feb 30).
    - Remove maxlength restrictions from inputs.
